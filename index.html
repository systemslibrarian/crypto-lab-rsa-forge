Why Textbook RSA Is Broken

  • Deterministic: Same plaintext always → same ciphertext. Attacker can build an encryption oracle and test known messages.
  • Homomorphic weakness: c₁ · c₂ mod n = Enc(m₁ · m₂) — ciphertexts can be multiplied to produce the encryption of the product.
  • Small message space: Low-entropy messages (like symmetric keys) are vulnerable to brute-force with an encryption oracle.
  • No ciphertext integrity: Any integer is a valid ciphertext — malleability attacks are trivial.
RFC 8017 (PKCS#1 v2.2), Section 7.1 — RSA-OAEP is the recommended encryption scheme. Textbook RSA is described only to explain the mathematical structure.

Small Exponent Attack — Håstad Broadcast

PADDING IS NON-NEGOTIABLE

If the same unpadded message m is sent to three recipients all using e=3, an attacker who intercepts all three ciphertexts can recover m using the Chinese Remainder Theorem and integer cube root — no private key needed.

Ref: Håstad, J. (1988). "Solving Simultaneous Modular Equations of Low Degree." SIAM Journal on Computing, 17(2), 336–341. Also: Coppersmith, D. (1997). "Small Solutions to Polynomial Equations, and Low Exponent RSA Vulnerabilities." Journal of Cryptology.

Why OAEP Destroys This Attack

  • Randomization: With OAEP, each recipient receives a different padded message. Even with the same underlying plaintext, the padded m values differ, so m³ values are not related by CRT.
  • Structure: The OAEP padding expands the message to fill the full modulus, so m is always close to n in size — m³ mod n wraps around the modulus multiple times.
  • Minimum key size: NIST SP 800-57 recommends e=65537 for all production RSA. Small e values (e=3, e=17) must only ever be used with proper padding.

Bleichenbacher PKCS#1 v1.5 Oracle

PKCS#1 v1.5 ENCRYPTION = AVOID

Bleichenbacher (1998) showed that a single-bit padding oracle — "is this ciphertext PKCS#1 v1.5 conformant?" — is enough to decrypt any RSA ciphertext via adaptive chosen-ciphertext queries. This broke SSL/TLS RSA key exchange and is still discovered in the wild today (ROBOT, 2017).

Ref: Bleichenbacher, D. (1998). "Chosen Ciphertext Attacks Against Protocols Based on the RSA Encryption Standard PKCS #1." CRYPTO 1998, LNCS 1462, pp. 1–12. Springer. The "million message attack."

Why TLS 1.3 Removed RSA Key Exchange Entirely

  • No forward secrecy: RSA key exchange uses the server's long-term private key to wrap the session key. Compromise of the private key retroactively decrypts all past traffic.
  • Bleichenbacher variants: ROBOT (2017) found 8 major TLS implementations still vulnerable 19 years after the original paper. The oracle is hard to eliminate from real implementations.
  • TLS 1.3 fix: RSA key exchange (RSA_PSK, RSA) removed. Only ephemeral Diffie-Hellman (ECDHE) allowed — provides forward secrecy by design.
  • PKCS#1 v1.5 encryption: RFC 8017 explicitly marks RSA-PKCS1_V1_5 encryption as legacy. Use RSA-OAEP for any new RSA encryption.
ROBOT (2017): Böck, Somorovsky, Young. "Return Of Bleichenbacher's Oracle Threat." USENIX Security 2018. Found in F5, Citrix, Palo Alto, and others.

RSA vs ECC vs Post-Quantum

QUANTUM-VULNERABLE

RSA security rests on the integer factorization problem. Shor's algorithm (1994) factors integers in polynomial time on a quantum computer, breaking all RSA key sizes. NIST's PQC standardization is complete — migration should begin now.

Shor's Algorithm — Why RSA Will Break

  • Classical factoring: Best classical algorithm (GNFS) runs in sub-exponential time — 264 operations for RSA-2048. Secure against classical computers.
  • Shor's algorithm: Reduces factoring to order-finding in polynomial time. Requires a fault-tolerant quantum computer with ~4,000 logical qubits for RSA-2048.
  • 2026 timeline: No quantum computer currently capable of breaking RSA-2048. But "harvest now, decrypt later" — adversaries store encrypted traffic today to decrypt with future hardware.
  • Migration urgency: NIST published FIPS 203/204/205 in August 2024. Systems with >10-year security requirements should migrate now.

RSA is in millions of systems in 2026
TLS certificates, SSH keys, code signing, email (S/MIME), and more all rely on RSA. Understanding RSA's failure modes — OAEP vs textbook, PKCS#1 v1.5 vulnerabilities, quantum threat — is essential for any engineer involved in the PQC migration.

Why This Matters

RSA remains one of the most widely deployed cryptographic algorithms on the planet in 2026 — found in TLS certificates, SSH keys, JWT signing, code signing, S/MIME email, and more. Yet real-world RSA deployments still fail in predictable ways: incorrect padding choices, small public exponents without proper padding, legacy PKCS#1 v1.5 encryption left in place. The Bleichenbacher attack — published 1998 — was still found in production TLS stacks in 2017. Understanding exactly how and why these attacks work is not academic; it is the difference between shipping secure systems and shipping exploitable ones.

The quantum migration has begun. Every system that uses RSA today needs a migration plan. ML-KEM (Kyber) and ML-DSA (Dilithium) are the replacements — explore them in the related demos below.
